Modelos de Estimación y Desarrollo de Software

1. ¿Qué son los Modelos de estimación?

Los Modelos de estimación son herramientas o métodos que se usan para calcular el tiempo, el esfuerzo y el costo que se necesitará para completar un proyecto. Es como cuando tratas de adivinar cuánto tiempo te llevará hacer la tarea de matemáticas; los modelos de estimación ayudan a hacer eso pero para proyectos más grandes, como desarrollar un programa de software.

2. Comparación de Modelos de Estimación

Modelo COCOMO (Constructive Cost Model): Este modelo usa una fórmula matemática para calcular el esfuerzo necesario en función del tamaño del software. Es muy bueno para proyectos grandes y complejos.

Método de Puntos de Función: Este modelo mide el tamaño del software basándose en las funcionalidades que ofrece al usuario. Es útil para proyectos donde la funcionalidad y la experiencia del usuario son muy importantes.

Comparación:

  • COCOMO: Es mejor para proyectos grandes y complejos donde se puede obtener una estimación precisa basada en el tamaño del software.
  • Puntos de Función: Es más adecuado cuando se quiere evaluar el proyecto desde la perspectiva del usuario y su experiencia, sin importar tanto el tamaño del software.

3. ¿A qué llamamos la estimación de un proyecto de software?

La estimación de un proyecto de software es el proceso de calcular cuánto tiempo, esfuerzo y recursos se necesitarán para completar el proyecto. Es como cuando intentas calcular cuánto tiempo te llevará estudiar para un examen, pero a una escala mucho mayor y con más detalles.

4. Principios Esenciales para el Desarrollo de Sistemas

  • Modularidad: Dividir un sistema en partes más pequeñas y manejables llamadas módulos.
  • Cohesión: Qué tan bien las tareas en un módulo están relacionadas entre sí.
  • Acoplamiento: Qué tan dependientes son los módulos entre sí.
  • Abstracción: Simplificar un sistema ocultando los detalles complejos y mostrando solo las partes esenciales.

Ejemplo de Modularidad: Supongamos que estamos desarrollando una aplicación de inventario. Podemos dividir la aplicación en módulos como "Gestión de Productos", "Gestión de Usuarios" y "Reportes". Cada módulo se encarga de una parte específica del sistema, lo que hace que sea más fácil de desarrollar y mantener.

5. Partes Interesadas (Stakeholders)

  • Usuarios Finales:
    • Rol: Utilizan la aplicación.
    • Responsabilidades: Proporcionar retroalimentación.
    • Expectativas: Que la aplicación sea fácil de usar y funcional.
  • Desarrolladores:
    • Rol: Crean la aplicación.
    • Responsabilidades: Escribir y mantener el código.
    • Expectativas: Tener especificaciones claras y herramientas adecuadas.
  • Gerentes de Proyecto:
    • Rol: Supervisan el proyecto.
    • Responsabilidades: Planificar, organizar y asegurar que el proyecto se complete a tiempo.
    • Expectativas: Que el proyecto se mantenga dentro del presupuesto y del cronograma.
  • Diseñadores de UX/UI:
    • Rol: Diseñan la interfaz de usuario.
    • Responsabilidades: Asegurarse de que la aplicación sea visualmente atractiva y fácil de usar.
    • Expectativas: Obtener retroalimentación clara y oportuna sobre sus diseños.
  • Clientes:
    • Rol: Financian el proyecto.
    • Responsabilidades: Proveer recursos y definir los requisitos.
    • Expectativas: Que el producto final cumpla con sus necesidades y objetivos.

6. Plan de Proyecto para un Sistema de Gestión de Inventario

  • Fase 1: Planificación
    • Definir requisitos
    • Establecer un cronograma
    • Asignar recursos
  • Fase 2: Análisis
    • Analizar requisitos
    • Crear especificaciones detalladas
  • Fase 3: Diseño
    • Diseñar la arquitectura del sistema
    • Diseñar la base de datos
  • Fase 4: Desarrollo
    • Programar los módulos del sistema
    • Integrar módulos
  • Fase 5: Pruebas
    • Probar el sistema
    • Corregir errores
  • Fase 6: Implementación
    • Desplegar el sistema
    • Capacitar a los usuarios
  • Fase 7: Mantenimiento
    • Resolver problemas
    • Actualizar el sistema

7. Normas y Estándares para el Desarrollo de Software

  • Convenciones de Nombres: Usar nombres descriptivos y consistentes para variables y funciones.
    • Ejemplo: En lugar de usar `x` o `temp`, usar `contadorDeUsuarios` o `fechaDeNacimiento`.
  • Formato del Código: Seguir un estilo consistente de sangría y espaciado.
    • Ejemplo: Usar cuatro espacios para cada nivel de sangría en el código.
  • Documentación del Código: Comentar el código para explicar su propósito y funcionamiento
